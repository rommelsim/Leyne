2026-07-02: owner decision — Android should use Material You (wallpaper-derived dynamic colour on Android 12+, seeded fallback below it), superseding the earlier "stay monochrome" rationale that was in `lib/theme.dart`. iOS is untouched and stays monochrome (different platform, see [[feedback_platform_design]]). Re-verify line numbers/details below before reusing, this is a snapshot; grep for symbols first.

**Wiring (lib/main.dart → lib/theme.dart):**
- `lib/main.dart`'s `LyneApp.build()` wraps the existing `ListenableBuilder(listenable: AppModel.shared, ...)` in a `DynamicColorBuilder` (from `package:dynamic_color`) — outer, so the platform-channel round trip to fetch the OS palette (effectively once per process) doesn't re-run on every `AppModel` notification. Passes `lightDynamic`/`darkDynamic` into `LyneTheme.light/dark.materialTheme(dynamicScheme: ...)`.
- `LyneTheme.materialTheme({ColorScheme? dynamicScheme})` in `lib/theme.dart`: uses `dynamicScheme ?? ColorScheme.fromSeed(seedColor: LyneSignal.meBlue, brightness: ...)`, calls `.harmonized()` (dynamic_color's `ColorScheme` extension), then `.copyWith(surface, onSurface, surfaceContainerHighest, outline, error, onError)` pinned back to the app's own `surface`/`fg`/`bg`/`line`/`crit`/`contrastFg` tokens. Only `primary`/`secondary` (and anything Flutter derives from them for free — `Switch` active track, `ChoiceChip` selected fill, both already M3-default-themed with no explicit `SwitchThemeData`/`ChipThemeData` in this codebase) follow the resolved dynamic/seeded scheme. NavigationBar's `indicatorColor` was also switched to `scheme.primary.withValues(alpha: ...)` (was a flat monochrome wash before).
- Seed colour is `LyneSignal.meBlue` (`0xFF3B82F6`, the existing "you are here" map-dot blue) — chosen because it was already an established brand-ish colour in the app, not a new invention.
- **Scope discipline**: dynamic colour tints CHROME + ACCENT only. `bg`/`surface`/`fg`/`line` etc. (read via `context.t` almost everywhere in the app) stay the static monochrome palette. MRT line colours (`MRTLine.color`), the new `LyneSeverity` enum, and `occupancyColor()` in `proximity.dart` (bus seat/standing/limited colours) are DATA and were explicitly excluded — none of them read from `ColorScheme`.
- `LyneTheme.accent`/`live` (used pervasively — LIVE dot, imminent-ETA numeral colour, FAB backgrounds, badges, route-timeline markers) DO go dynamic, but only through `context.t`: `LyneThemeContext.t` now resolves `Theme.of(context).colorScheme.primary` and returns `base.withAccent(primary)` (new `LyneTheme.withAccent(Color)` copy method) instead of the raw static `LyneTheme.light`/`dark` singleton. The static fields on `LyneTheme.light`/`.dark` themselves stay the monochrome fallback value — only the context-resolved copy is dynamic. `onAccent` was left as the existing `isDark ? black : white` heuristic (not luminance-computed) — this is safe because M3-generated primaries (both `ColorScheme.fromSeed` and the OS CorePalette) always use tone ~40 in light / ~80 in dark, which is exactly the tone range that heuristic already assumes.

**LyneSeverity token (lib/theme.dart, new enum near `LyneSignal`/`MRTLine`):**
`enum LyneSeverity { normal(Colors.green), warning(Colors.orange), critical(Colors.red), unknown(Color.fromRGBO(128,128,128,0.35)) }` — centralises what were ~20 raw `Colors.orange`/`green`/`red` literals for MRT line disruption status and station/line `CrowdLevel` across `soft_mrt_screen.dart`, `soft_mrt_line_screen.dart`, `soft_mrt_station_screen.dart`, `soft_alerts_screen.dart`. Cross-mode like `LyneSignal`/`MRTLine` (same value in light/dark), never Material-You-driven (data, not chrome). This is a DIFFERENT axis from `occupancyColor()` in `proximity.dart` (LTA `Load` seat/standing colours) — that one was explicitly left untouched.

**mrt_map_screen.dart inverse-panel fix:** now uses `final inv = LyneTheme.light;` (not `context.t`) as the source for `contrast`/`contrastFg`, because `LyneTheme.light.contrast` is ALREADY the dark-panel-on-light-mode value (near-black `#111111`) by the token's own design intent ("inverse panel"). Using `context.t` there would flip to white in dark app-mode, breaking the screen's intentional always-dark immersive chrome. `_MapFallback`'s translucent panel/button backgrounds became `t.contrastFg.withValues(alpha: 0.08/0.15)` instead of raw white-alpha literals.

**Toolchain gotcha — do not reuse `Theme.SplashScreen`/`postSplashScreenTheme` on this machine's SDK:**
The installed Android SDK here is `compileSdk 36` / `build-tools 36.0.0` (`~/Library/Android/sdk/platforms/android-36`). `aapt2 dump resources android.jar` confirms `windowSplashScreenBackground`/`windowSplashScreenAnimatedIcon`/`windowSplashScreenIconBackgroundColor` ARE public and linkable, but `android:style/Theme.SplashScreen` and `android:attr/postSplashScreenTheme` (the combo every official guide/tutorial uses) are NOT in this SDK's public compile-time resource table — referencing either fails `aapt2` resource linking (`resource android:style/Theme.SplashScreen not found` / `style attribute 'android:attr/postSplashScreenTheme' not found`), even though official Android docs describe them as current API. Root cause not fully confirmed (plausibly an Android 16/API 36 SDK stub-jar omission — there's a `splashScreenTheme` PUBLIC attr at the same ID range with a suspiciously adjacent number, and `windowSplashScreenBehavior` is marked `STAGED_ID`, suggesting API 36 reworked this surface) — re-check if a newer SDK platform revision is installed later.
Workaround used (`android/app/src/main/res/values-v31/styles.xml` + `values-night-v31/styles.xml`): layer `windowSplashScreenBackground`/`windowSplashScreenAnimatedIcon` directly onto the SAME parent the pre-12 `LaunchTheme` already uses (`Theme.Light.NoTitleBar` / `Theme.Black.NoTitleBar`), skip `Theme.SplashScreen` entirely, and don't set any post-dismiss theme-switch attribute at all — because `FlutterActivity.onCreate()` unconditionally calls `switchLaunchThemeForNormalTheme()` (confirmed by reading the engine source at `.../flutter/shell/platform/android/io/flutter/embedding/android/FlutterActivity.java`), which already does that exact job on every API level via the pre-existing `io.flutter.embedding.android.NormalTheme` manifest meta-data. This is Flutter-app-specific — a non-Flutter native Android app on this same SDK would still need a real fix (or an older `compileSdk`) for the missing `postSplashScreenTheme`.
